Modelo Relacional

Apuntes completos + ejercicios resueltos · Bases de Datos · ETSI Informáticos UPM · Antonio J. Díaz Honrubia

1. Introducción al Modelo Relacional

La estructura fundamental del Modelo Relacional es la relación. Matemáticamente, una relación R de orden n definida sobre conjuntos A₁, A₂, …, Aₙ es un subconjunto del producto cartesiano A₁ × A₂ × … × Aₙ.

Grado / Orden — número de atributos (columnas) de la relación.
Cardinalidad — número de tuplas (filas) de la relación.
Tupla — cada uno de los elementos de la relación.

Ejemplo concreto

Sean A = {a, b, c} y B = {1, 2}. Dos relaciones de orden 2 son:

R₁ = {(a,1), (a,2), (c,2)} → cardinalidad 3 R₂ = {(a,1), (b,1)} → cardinalidad 2

La tupla (a,1) pertenece a R₁. El grado de ambas es 2, porque tienen dos atributos (A y B).

Propiedades fundamentales de una relación

Una relación es un conjunto, por lo tanto:
• No puede haber tuplas duplicadas.
• No hay un orden definido sobre las tuplas (son un conjunto, no una lista).

Representación tabular

La forma habitual de visualizar una relación es como una tabla:

ABC
a5y
c10x
b1y

El número de columnas es el grado; el número de filas, la cardinalidad; cada columna corresponde a un atributo.

2. Atributos y dominios

Atributo — cada "dimensión" o propiedad de la relación (columna).
Dominio — conjunto de valores posibles para un atributo. Puede ser finito o infinito.
Universo T — conjunto de todos los atributos de un problema. Siempre finito.
Descriptor X — cualquier subconjunto de T. Se escribe concatenado: X = AC significa X = {A, C}.

Tipos de dominios

Dominios comunes: integer, real, date, datetime, character, string. El valor especial NULL representa "desconocido" o "sin valor". Ojo: 0 y la cadena vacía "" no son NULL.

Reglas sobre atributos

• No puede haber dos atributos distintos con el mismo nombre en la misma relación.
• Atributos distintos pueden compartir el mismo dominio.
• La elección del nombre del atributo es independiente de su dominio.

3. Dependencias funcionales

Si X e Y son descriptores y, dado un valor de X, el valor de Y queda unívocamente determinado, existe la dependencia funcional X → Y.
Se lee: "X implica Y" o "Y depende funcionalmente de X".

Dependencia total vs. parcial

La dependencia X → Y es total si no existe ningún subconjunto propio Z ⊂ X tal que Z → Y. En caso contrario es parcial (Y depende de una parte de X, no de X completo).

Esquema de una relación

El esquema R(T, L) es el par formado por el universo de atributos T y el conjunto de dependencias L. La relación universal R(T, L) incluye todos los atributos del problema.

Axiomas de Armstrong

Permiten deducir nuevas dependencias a partir de las conocidas. Son sólidos (todo lo deducido es correcto) y completos (todo lo que se puede deducir se puede deducir con ellos).

Reflexividad (primario)
Si Y ⊆ X, entonces X → Y
En particular: X → X
Aumento (primario)
Si X → Y, entonces XZ → YZ
para cualquier descriptor Z
Transitividad (primario)
Si X → Y e Y → Z,
entonces X → Z
Proyectividad (derivado)
Si X → YZ, entonces X → Y
Aditividad (derivado)
Si X → Y y Z → W,
entonces XZ → YW
Unión (derivado)
Si X → Y y X → Z,
entonces X → YZ

4. Claves de una relación

Dada R sobre T, un descriptor K es superclave si K → T.
Si además K es el subconjunto mínimo posible que determina T, es clave candidata.
De entre todas las candidatas, se elige una como clave primaria; las demás son claves alternativas.
Atributo primario / clave — atributo que pertenece a alguna clave candidata.
Integridad de entidad — no se puede insertar una tupla si se desconoce el valor de algún atributo primario (clave primaria nunca puede ser NULL).
Claves foráneas (foreign keys) — restricciones de integridad referencial: el valor de ciertos atributos en una relación debe existir como clave primaria en otra relación. Garantizan la consistencia entre tablas.

5. Álgebra Relacional

El álgebra relacional define un conjunto de operadores que, aplicados a relaciones, producen nuevas relaciones (relaciones derivadas). Es la base teórica del lenguaje SQL.

Existe también el cálculo relacional (lógica de predicados), equivalente al álgebra pero con distinta notación. Nosotros usaremos el álgebra.

6. Principios de diseño de bases de datos

El objetivo del diseño relacional es minimizar redundancia y anomalías (de inserción, borrado y actualización).

6.1 Cierre de un descriptor

El cierre X⁺ de un descriptor X respecto al conjunto de dependencias L es el mayor conjunto de atributos que X puede determinar usando L y los axiomas de Armstrong.

Propiedades que definen X⁺:

1) X ⊆ X⁺  |  2) X → X⁺  |  3) No existe Y ⊋ X⁺ tal que X → Y

Algoritmo para calcular X⁺

  1. Inicializar: X⁰ := X
  2. Repetir: Xⁱ⁺¹ := Xⁱ ∪ Y para toda dependencia (Z → Y) ∈ L con Z ⊆ Xⁱ
  3. Parar cuando Xⁱ⁺¹ = Xⁱ. Ese Xⁱ es X⁺.

Ejemplo (de los apuntes)

T = {A,B,C,D,E,F}, L = {A→BC, C→D, AC→E, AB→C, F→A}, X = ABE

X⁰ = ABE X¹ = ABE ∪ BC (aplicando A→BC, pues A ⊆ X⁰) = ABCE X² = ABCE ∪ D (aplicando C→D, pues C ⊆ X¹) = ABCED X³ = ABCED (ninguna regla añade nada nuevo) X³ = X² → X⁺ = ABCED

6.2 Recubrimiento no redundante

M es recubrimiento no redundante de L si:
1) L⁺ = M⁺ (equivalentes)
2) Toda dependencia de M tiene implicado simple (solo un atributo a la derecha)
3) En todos los implicantes no hay atributos superfluos
4) En M no hay dependencias redundantes

Algoritmo (3 pasos)

  1. Obtener implicados simples: aplicar proyectividad hasta que todo implicado sea un solo atributo. (A→BC se convierte en A→B y A→C).
  2. Eliminar atributos superfluos en implicantes: para cada atributo X en un implicante compuesto XZ→Y, comprobar si Z→Y es deducible sin usar dicha dependencia. Si el cierre de X en L sin esa dependencia ya contiene a Y, el atributo es superfluo.
  3. Eliminar dependencias redundantes: para cada dependencia X→Y, comprobar si el cierre de X en L sin esa dependencia ya contiene a Y.

6.3 Algoritmo para calcular una clave

Dado un esquema R(T, L) con L no redundante, definimos 4 conjuntos:

I — atributos que aparecen solo en implicantes (izquierda) de L.
D — atributos que aparecen solo en implicados (derecha) de L.
ID — atributos que aparecen a ambos lados de dependencias en L.
N — atributos que no aparecen en ninguna dependencia de L.
Z = I ∪ N → es el núcleo (intersección de todas las claves candidatas).

Si Z⁺ = T, entonces Z es la clave única. Si no, se añaden atributos de ID hasta que el cierre sea T. Cada subconjunto mínimo que alcanza T es una clave candidata.

Ejemplo: clave única

T={A,B,C,D,E,F}, L={A→B, C→D, A→E, A→C, F→A}

I = {F} (solo aparece a la izquierda) D = {B,D,E} (solo aparecen a la derecha) ID = {A,C} (aparecen a ambos lados) N = {} Z = I ∪ N = {F} Z⁺ = F⁺ = ABCDEF = T → F es clave única.

7. Normalización de bases de datos

La normalización es el proceso de reorganizar un esquema relacional para eliminar redundancia, reducir costes de actualización y proteger la integridad de los datos.

7.1 Formas Normales

Forma normalRequisito adicional sobre la anterior
1FN Atributos atómicos, clave primaria definida, sin duplicados, sin valores nulos en clave, todos los no-clave dependen funcionalmente de la clave.
2FN Está en 1FN. Ningún atributo no-clave depende parcialmente de una clave (solo se viola si la clave es compuesta).
3FN Está en 2FN. Ningún atributo no-clave depende transitivamente de una clave. En resumen: todo atributo no-clave depende de una clave, la clave completa y nada más que la clave.
FNBC Versión más fuerte de 3FN: todo implicante de cualquier dependencia no trivial debe ser superclave (clave candidata). Si A→B, entonces A debe ser clave.
4FN Está en FNBC. No existen dependencias multivaluadas no triviales independientes con el mismo implicante.
5FN Está en 4FN y no puede descomponerse sin pérdida de información.

Anomalías

Anomalía de inserción — no se pueden insertar datos de una entidad sin conocer datos de otra entidad ligada por la clave.
Anomalía de borrado — al eliminar la única tupla que contiene cierta información, se pierde esa información colateralmente.
Anomalía de actualización — modificar un dato repetido en varias filas puede dejar la BD inconsistente si solo se actualiza en algunas.

7.2 Algoritmo de Ullman (descomposición en 3FN)

El algoritmo de Ullman descompone un esquema R(T,L) en subesquemas R₁,…,Rₙ tal que: cada Rᵢ está en 3FN, se preservan las dependencias y es lossless-join (la relación original se puede reconstruir haciendo join de los subesquemas).

  1. Obtener el recubrimiento no redundante de L (3 pasos del apartado anterior).
  2. Calcular una clave del esquema original con el algoritmo I/D/ID/N.
  3. Agrupar dependencias con el mismo implicante en una misma relación: Rᵢ(Tᵢ, Lᵢ) donde Tᵢ es el implicante + sus implicados.
  4. Eliminar subesquemas redundantes: si Tᵢ ⊆ Tⱼ para algún j≠i, eliminar Rᵢ.
  5. Garantizar la clave: si ningún subesquema contiene la clave calculada, añadir un nuevo subesquema R₀ con los atributos de dicha clave.

Ejercicio 3 — Recubrimiento no redundante y Ullman

Datos T = {P,S,D,A,Z,X,V,E}   L = {P→SDA, D→Z, AZX→V, SV→E, PX→E}
a) Recubrimiento no redundante
Paso 1: implicados simples (proyectividad)
P→SDA se descompone en: P→S, P→D, P→A Resto no cambia. L₁ = {P→S, P→D, P→A, D→Z, AZX→V, SV→E, PX→E}
Paso 2: eliminar atributos superfluos en implicantes
AZX→V: ¿Es A superfluo? Cierre de ZX en L₁\{AZX→V}: ZX⁺ = ZX (Z y X no aparecen en ningún implicante del resto) A ∉ ZX⁺ → A no es superfluo. ¿Es Z superfluo? Cierre de AX en L₁\{AZX→V}: AX⁺: A→S, A→D, A→A; D→Z; AX⁺ = ASDZ (añadimos S,D,Z) V ∉ AX⁺ → Z no es superfluo. ¿Es X superfluo? Cierre de AZ en L₁\{AZX→V}: AZ⁺: A→S,D,A; D→Z; AZ⁺ = ASZD V ∉ AZ⁺ → X no es superfluo. PX→E: ¿Es P superfluo? Cierre de X en L₁\{PX→E}: X⁺ = X → E ∉ X⁺ → P no es superfluo. ¿Es X superfluo? Cierre de P en L₁\{PX→E}: P⁺ = PSDAZ (por P→S,P→D,P→A,D→Z) E ∉ P⁺ → X no es superfluo. → Sin cambios. L₂ = L₁.
Paso 3: eliminar dependencias redundantes
¿P→S redundante? Cierre de P en L₂\{P→S}: P⁺ = PDA Z (P→D, P→A, D→Z) → S ∉ P⁺ → NO redundante. ¿P→D redundante? Cierre de P en L₂\{P→D}: P⁺ = PSA → D ∉ P⁺ → NO redundante. ¿P→A redundante? Similar → NO redundante. ¿SV→E redundante? Cierre de SV en L₂\{SV→E}: SV⁺ = SV → E ∉ SV⁺ → NO redundante. ¿PX→E redundante? Cierre de PX en L₂\{PX→E}: PX⁺: P→S,D,A; D→Z; AZX→V; SV→E PX⁰=PX; +S,D,A → PXSDA; +Z(de D) → PXSDAZ; +V(de AZX,X∈PX) → PXSDAZV; +E(de SV) → PXSDAZVE E ∈ PX⁺ → PX→E ES REDUNDANTE. Eliminar. Recubrimiento no redundante M: M = {P→S, P→D, P→A, D→Z, AZX→V, SV→E}

PX→E es redundante porque PX ya implica S y V (vía P→S y AZX→V con X en el implicante), y luego SV→E da E. Por tanto podemos prescindir de PX→E.

b) Claves candidatas
Clasificar atributos con M

Ejercicio 5 — Equivalencia de esquemas R(T,L₁) y S(T,L₂)

Datos T={A,B,C,D,E,F}    L₁={A→C, C→B, D→B, D→E, C→D}    L₂={A→CD, BC→A, D→BE, C→D}
Método: dos esquemas son equivalentes si L₁⁺ = L₂⁺, es decir, L₁ ⊆ L₂⁺ y L₂ ⊆ L₁⁺
Verificar que L₁ ⊆ L₂⁺ (cada dep de L₁ se deduce de L₂)
A→C: A⁺ con L₂: A→CD (de A→CD) → C ∈ A⁺ ✓ C→B: C⁺ con L₂: C→D (de C→D); D→BE → B ∈ C⁺ ✓ D→B: D⁺ con L₂: D→BE → B ∈ D⁺ ✓ D→E: D→BE → E ∈ D⁺ ✓ C→D: C→D directamente en L₂ ✓ → Todas las dependencias de L₁ son deducibles de L₂. L₁ ⊆ L₂⁺.
Verificar que L₂ ⊆ L₁⁺ (cada dep de L₂ se deduce de L₁)
A→CD: A⁺ con L₁: A→C; C→B; C→D → ABCD ⊇ {C,D} ✓ BC→A: BC⁺ con L₁: C→B (ya tenemos B); C→D; D→B,E; A→C(necesitamos A a la izq) BC⁰=BC; C→B(ya); C→D → BCD; D→B,E → BCDE A ∉ BCDE → ¿podemos deducir BC→A? Verifiquemos si hay algún camino: desde B y C no hay ninguna dependencia que dé A. A ∉ BC⁺ con L₁. BC→A NO es deducible de L₁.
Conclusión

Los esquemas NO son equivalentes. BC→A ∈ L₂ no puede deducirse de L₁, por lo que L₂⁺ ≠ L₁⁺.

La dependencia BC→A implica que conocer el valor de B y C determina unívocamente A. En L₁ no hay ninguna cadena de dependencias que permita deducir esto. En L₁, A solo aparece como implicante (A→C), nunca como implicado.
